# Mobile app 방향성

이 폴더는 **Android(Kotlin)** 및 **iOS(Swift)** 네이티브 앱을 제작하기 위한
공통 가이드와 플랫폼별 준비 사항을 정리합니다.

## 목표
- Android는 Kotlin + Jetpack Compose로 구현
- iOS는 Swift + SwiftUI로 구현
- 데이터는 현재 `data/hololive_ocg.sqlite`를 **read-only**로 내장하고,
  앱 업데이트 시 교체하는 방식을 기본으로 합니다.
- 기존 DB 스키마는 변경하지 않습니다.

## 네이티브 vs Flet 차이 (요약)
- **UI/성능**: 네이티브는 플랫폼 전용 UI/렌더링으로 성능과 일관성이 높고, Flet은 Python 기반 크로스플랫폼 UI로 빠른 개발이 장점입니다.
- **플랫폼 기능 접근**: 네이티브는 카메라/알림/위젯 등 시스템 기능 접근이 용이하며, Flet은 플러그인/브리지 필요합니다.
- **배포/운영**: 네이티브는 각 스토어 정책/빌드 체인을 따르며, Flet은 데스크톱 중심 워크플로우에 가깝습니다.
- **공유 코드**: Flet은 Python 로직을 공통으로 쓰기 쉬우나, 네이티브는 플랫폼별 구현이 필요합니다.

## 해야 할 것 (모바일 전환 체크리스트)
1. **요구사항 정리**: Flet 앱 기능 목록 → 모바일 필수 기능/우선순위 재정의
2. **데이터 전략**: `data/hololive_ocg.sqlite` 번들 방식 확정 (읽기 전용)
3. **아키텍처 설계**: Android (MVVM + Compose), iOS (MVVM + SwiftUI) 구조 확정
4. **기본 화면 구현**: 카드 리스트/상세/덱 빌더의 MVP 범위 결정
5. **검색/필터**: 성능 기준에 맞는 SQL 쿼리 정의
6. **이미지 전략**: 로컬/원격 로딩 및 캐시 정책 결정
7. **빌드/배포**: 스토어 등록/서명/CI 파이프라인 준비

## 공통 데이터 계약 (요약)
- DB 파일: `data/hololive_ocg.sqlite`
- 주요 테이블: `cards`, `prints`, `card_texts_ko` 등 (스키마 변경 금지)
- 앱에서는 읽기 전용으로 접근하며, 앱 번들 내 DB를 복사해 사용합니다.
